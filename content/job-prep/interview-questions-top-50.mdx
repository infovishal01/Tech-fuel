## 📊 Data Structures & Algorithms (20 Questions)

### Arrays & Strings

#### 1. Two Sum
**Problem**: Given an array of integers `nums` and an integer `target`, return indices of the two numbers such that they add up to `target`.

**Difficulty**: Easy  
**LeetCode**: [#1](https://leetcode.com/problems/two-sum/)  
**Companies**: Almost all  
**Frequency**: ⭐⭐⭐⭐⭐

**Solution Approach**:
- Brute Force: O(n²) - Check all pairs
- Optimized: O(n) - Use hash map to store seen values

```python
def two_sum(nums, target):
    seen = {}
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    return []
```

## 🎯 Object-Oriented Programming (5 Questions)

#### 31. OOP Principles
**Question**: Explain the four pillars of Object-Oriented Programming.

**Key Points**:
- **Encapsulation**: Bundling data and methods
- **Inheritance**: Creating new classes from existing ones
- **Polymorphism**: Same method name, different implementations
- **Abstraction**: Hiding complex implementation details

**Example**:
```java
// Encapsulation
class BankAccount {
    private double balance;  // Private data
    
    public void deposit(double amount) {  // Public method
        if (amount > 0) balance += amount;
    }
}

// Inheritance
class Animal {
    void eat() { System.out.println("Eating..."); }
}

class Dog extends Animal {
    void bark() { System.out.println("Barking..."); }
}

// Polymorphism
interface Shape {
    double area();
}

class Circle implements Shape {
    double radius;
    double area() { return Math.PI * radius * radius; }
}

class Square implements Shape {
    double side;
    double area() { return side * side; }
}

// Abstraction
abstract class Vehicle {
    abstract void move();  // What it does
    // Implementation hidden
}
```

#### 32. SOLID Principles
**Question**: Explain SOLID principles in OOP.

**Key Points**:
- **S**ingle Responsibility Principle (SRP)
- **O**pen/Closed Principle (OCP)
- **L**iskov Substitution Principle (LSP)
- **I**nterface Segregation Principle (ISP)
- **D**ependency Inversion Principle (DIP)

#### 33. Design Patterns
**Question**: Explain common design patterns with examples.

**Key Patterns**:
- **Singleton**: Ensure only one instance exists
- **Factory**: Create objects without specifying exact class
- **Observer**: Notify dependents of state changes
- **Strategy**: Define a family of algorithms
- **Builder**: Separate construction from representation
- **Decorator**: Add behavior dynamically

#### 34. Abstract Class vs Interface
**Question**: What's the difference between an abstract class and an interface?

**Key Differences**:
| Feature | Abstract Class | Interface |
|---------|---------------|-----------|
| Methods | Can have implementation | Only declarations |
| Variables | Can have instance variables | Only constants |
| Constructors | Can have constructors | No constructors |
| Inheritance | Single inheritance | Multiple inheritance |
| Access | Can have any access modifier | Only public |

#### 35. Composition vs Inheritance
**Question**: When to use composition over inheritance?

**Key Points**:
- **Inheritance**: "Is-a" relationship
- **Composition**: "Has-a" relationship
- **Favor composition** over inheritance (more flexible)
- **Inheritance** can lead to complex hierarchies
- **Composition** allows changing behavior at runtime

## 🗄️ Databases (5 Questions)

#### 36. SQL vs NoSQL
**Question**: What are the differences between SQL and NoSQL databases?

**Key Differences**:
| Feature | SQL | NoSQL |
|---------|-----|-------|
| Data Model | Relational (tables) | Non-relational (documents, key-value, etc.) |
| Schema | Fixed schema | Dynamic schema |
| ACID | Full ACID compliance | eventual consistency (BASE) |
| Scalability | Vertical | Horizontal |
| Query Language | SQL | Varies (MongoDB Query Language, etc.) |
| Examples | MySQL, PostgreSQL | MongoDB, Cassandra, Redis |

#### 37. Indexing
**Question**: How do database indexes work? What are the trade-offs?

**Key Points**:
- **What**: Data structure that improves query speed
- **Types**: B-tree (most common), Hash, Bitmap, Full-text
- **Trade-offs**:
  - ✅ Faster SELECT queries
  - ❌ Slower INSERT/UPDATE/DELETE
  - ❌ More storage space
  - ❌ Maintenance overhead
- **When to use**: Columns frequently used in WHERE, JOIN, ORDER BY

#### 38. Normalization
**Question**: Explain database normalization with examples.

**Key Points**:
- **1NF**: Atomic values, no repeating groups
- **2NF**: No partial dependencies (full dependency on PK)
- **3NF**: No transitive dependencies
- **BCNF**: Stronger than 3NF
- **Trade-off**: Normalization vs Denormalization

#### 39. Transactions & ACID
**Question**: What is ACID in database transactions?

**Key Points**:
- **Atomicity**: All or nothing (all operations succeed or none do)
- **Consistency**: Database remains in consistent state
- **Isolation**: Concurrent transactions don't interfere
- **Durability**: Committed transactions persist

**Isolation Levels**:
1. Read Uncommitted (dirty reads allowed)
2. Read Committed (prevents dirty reads)
3. Repeatable Read (prevents non-repeatable reads)
4. Serializable (highest isolation)

#### 40. CAP Theorem
**Question**: Explain the CAP theorem.

**Key Points**:
- **Consistency**: All nodes see the same data
- **Availability**: Every request receives a response
- **Partition Tolerance**: System continues despite network partitions
- **Theorem**: Can only guarantee 2 out of 3
- **CP Systems**: Consistency + Partition Tolerance (MongoDB, HBase)
- **AP Systems**: Availability + Partition Tolerance (Cassandra, CouchDB)
- **CA Systems**: Consistency + Availability (only when no partitions - rare)

## 💻 Operating Systems (5 Questions)

#### 41. Processes vs Threads
**Question**: What's the difference between processes and threads?

**Key Differences**:
| Feature | Process | Thread |
|---------|---------|--------|
| Definition | Program in execution | Lightweight process |
| Memory | Separate address space | Shared address space |
| Creation | Slow (requires OS) | Fast |
| Communication | IPC required | Direct memory access |
| Overhead | High | Low |
| Stability | One crash doesn't affect others | One crash can bring down all |

#### 42. Memory Management
**Question**: How does the OS manage memory?

**Key Concepts**:
- **Virtual Memory**: Gives each process its own address space
- **Paging**: Divide memory into fixed-size pages
- **Segmentation**: Divide memory into variable-size segments
- **Demand Paging**: Load pages only when needed
- **Page Replacement**: LRU, FIFO, Optimal, etc.
- **Thrashing**: Too much page faulting

#### 43. Process Synchronization
**Question**: Explain process synchronization mechanisms.

**Key Mechanisms**:
1. **Mutex (Mutual Exclusion)**: Lock for critical sections
2. **Semaphore**: Integer counter for resource access
3. **Monitor**: High-level synchronization construct
4. **Condition Variables**: Threads wait for conditions
5. **Barriers**: Synchronization point for threads

**Classic Problems**:
- Producer-Consumer
- Dining Philosophers
- Readers-Writers

#### 44. Deadlocks
**Question**: What are deadlocks? How to prevent and handle them?

**Key Points**:
- **Definition**: Two or more processes waiting indefinitely for each other
- **Conditions (Coffman Conditions)**:
  1. Mutual Exclusion
  2. Hold and Wait
  3. No Preemption
  4. Circular Wait
- **Prevention**: Break one of the conditions
- **Avoidance**: Banker's algorithm
- **Detection**: Wait-for graph
- **Recovery**: Process termination or resource preemption

#### 45. Scheduling Algorithms
**Question**: Explain different CPU scheduling algorithms.

**Key Algorithms**:
1. **FCFS (First-Come, First-Served)**: Non-preemptive, simple but can have convoy effect
2. **SJF (Shortest Job First)**: Non-preemptive, optimal for average waiting time
3. **SRTF (Shortest Remaining Time First)**: Preemptive SJF
4. **Round Robin**: Time quantum, preemptive, fair but high overhead
5. **Priority Scheduling**: Preemptive/non-preemptive, can lead to starvation
6. **Multilevel Queue**: Different queues with different priorities
7. **Multilevel Feedback Queue**: Can move between queues

## 🧠 Behavioral & Soft Skills (5 Questions)

#### 46. Tell Me About Yourself
**Question**: Walk me through your background.

**Structure (1-2 minutes)**:
1. **Present**: Current role, company, key responsibilities (30 seconds)
2. **Past**: Previous experiences, education (30 seconds)
3. **Future**: Why this role, why this company (30 seconds)

**Example**:
```
"I'm currently a Software Engineer at Tech Corp, where I work on backend services 
using Node.js and MongoDB. Before this, I completed my Computer Science degree from 
State University, where I developed a strong foundation in algorithms and data 
structures. I'm particularly passionate about building scalable systems and solving 
complex problems. I'm excited about this opportunity at your company because I 
admire your work in AI and would love to contribute to your mission of making 
technology more accessible."
```

**Pro Tips**:
- ✅ Be concise and relevant
- ✅ Tailor to the job description
- ✅ Show enthusiasm
- ❌ Don't recite your entire resume
- ❌ Don't be too personal

#### 49. Conflict Resolution
**Question**: Tell me about a time you had a conflict with a teammate.

**Structure (STAR Method)**:
1. **Situation**: Briefly describe the context
2. **Task**: What needed to be accomplished
3. **Action**: What YOU did to resolve the conflict
4. **Result**: Positive outcome

**Example**:
```
"In my previous project, my teammate and I had a disagreement about the architecture 
for a new feature. I wanted to use a microservices approach, while they preferred a 
monolithic design. (Situation)

We needed to implement this feature quickly to meet a client deadline, and the 
decision was blocking our progress. (Task)

Instead of arguing, I suggested we each create a prototype of our approach and 
present the pros and cons to the team. I spent the weekend building my version and 
preparing a detailed comparison. During the meeting, we realized that a hybrid approach 
would work best. (Action)

We ended up delivering the feature on time, and it became one of our most stable 
and maintainable features. The experience also improved our working relationship, as 
we both learned to appreciate each other's perspectives. (Result)"
```

**Pro Tips**:
- ✅ Show that you can handle conflict professionally
- ✅ Focus on the positive resolution
- ✅ Highlight what you learned
- ❌ Don't blame the other person
- ❌ Don't make up a story
